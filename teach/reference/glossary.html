Build a docs Skill · Reference

Course glossary

Plain-language definitions for every term in the course. We keep these stable: once a word is defined here, every lesson uses it the same way. Skim it before a lesson, or come back when a word trips you up.

Grounding
Answering from a real source you fetched, not from the model's memory. A grounded answer can show you the exact page it used. First seen: Lesson 1
Harness
Everything around the model: instructions, tools, the docs it can reach, and the tests that check it. Captured by Agent = Model + Harness. A skill is part of the harness. First seen: Lesson 1
Skill
A small package you ship to an agent — a SKILL.md file plus optional helper scripts and data — that teaches it how to do one job well (here: search and read the Teleport docs). It changes the harness, not the model. First seen: Lesson 1
SKILL.md
The main file of a skill. It has a short YAML header (a name and a description that help the agent decide when to use the skill) and a body that teaches the steps to follow. First seen: Lesson 1 (full detail in Lesson 2)
search → fetch (the protocol)
The two-move loop at the heart of the skill. Search finds the right doc page for a question; fetch pulls that live page's text so the agent answers from real words. We borrowed this shape from the AWS Docs MCP server. Later it grows a third optional move, related (sibling pages). First seen: Lesson 1 (full detail in Lesson 2)
llms.txt
A plain-text index a website publishes for LLMs, at a known URL (e.g. goteleport.com/docs/llms.txt). It lists every page as - [Title](url.md): description under ## Section headings. We search this instead of crawling the site. First seen: Lesson 2
Manifest
The list of pages inside an llms.txt — title + URL + one-line description, one entry per page. This is what our search move reads. First seen: Lesson 2
Markdown twin
The clean markdown version of a doc page, served when you add .md to its URL. No HTML, no nav to strip; starts with a {"token_count": N} line. This is what our fetch move pulls. First seen: Lesson 2
related (the third move)
An optional follow-up to search → fetch: list the sibling pages in the same manifest section, for when one page isn't enough. First seen: Lesson 2
Progressive disclosure / progressive discovery
The harness surfaces relevant info in small bits, and the LLM decides whether to dive deeper. Read cheap one-line entries to pick a page, then spend tokens on the full page only for the one you picked — go only as deep as the question needs. Applies to the skill file itself: a short SKILL.md index that references out to advanced topics and workflows, instead of dumping everything up front. Keeps the agent fast and its context small. (Not to be confused with Agile iteration, the build → test → improve loop.) First seen: Lesson 2
Title-only search
Searching only the manifest line — a page's title and one-line description. Cheap and fast, but blind to anything in the page body. What we start with. First seen: Lesson 3
Full-text (content) search
Searching the entire page body, not just its label. Finds error strings, config keys, file names, and CLI flags that live deep in a page. What we want. First seen: Lesson 3
Body-only query
A question whose key words appear only in a page's body, never in its title or description. The exact case title-only search misses and full-text search catches. First seen: Lesson 3
Complete miss (rank 0)
When the correct page shares no words with the label the search reads, so it never matches — the right answer scores zero. First seen: Lesson 3
Eval
A test for an agent or skill: a question plus what a good answer must contain (and must not contain). Run it to check the skill still works. First seen: Lesson 4
Benchmark
Running many evals to compare two approaches by a number — e.g. title-only vs full-text, scored as wins. Turns "better" from opinion into evidence. First seen: Lesson 4
Regression bar
An eval kept forever to stop a fixed bug from coming back. Every past failure becomes a permanent assertion. First seen: Lesson 4
Red → green
Test-first discipline: write the failing test first (red), then build the fix that makes it pass (green) — and keep the test as a regression bar. In this project, the body-only evals were written before the better engine existed. First seen: Lesson 4
Agile iteration (short feedback loops)
Ship a minimal design, put it to the test, improve — in short rounds. Each round sources more accurate data than the last: borrow the index → search the body → rank by density (BM25) → index code blocks. Pairs with red → green: every round's fix is locked by a test. (This is NOT progressive discovery — see below.) First seen: Lesson 4
Worth-it lens (ROI)
The habit of weighing ancillary work (indexing, evals, tuning) against just doing the real task — times how many agents will reuse the skill. Sometimes the honest answer is "don't build it." First seen: Lesson 4
TF-IDF
A way to rank pages by counting words — "just arithmetic," no model needed. weight = TF × IDF, summed over the query's words. Chosen because it ships as a static file, not because it retrieves best. First seen: Lesson 5
Term frequency (TF)
How often a word appears in a page. A word used a lot in a page is probably important to that page. First seen: Lesson 5
Inverse document frequency (IDF)
How rare a word is across all pages: log(total pages / pages with the word). Rare words score high (strong signal); words in nearly every page score low. First seen: Lesson 5
Embeddings / vector search
Turning text into lists of numbers (vectors) with a model, then matching by meaning. Powerful, but to search you need the same model at query time — so it can't ship in a static file; it needs a hosted server. First seen: Lesson 5 (weighed in Lesson 9)
Build-time vs query-time
Do the heavy math once when you build the index (download pages, compute weights, save JSON); the shipped skill does only cheap lookups and addition at query time — no model, no GPU. First seen: Lesson 5
Static index
The search data shipped as a plain file (here a ~4 MB search-index.json), so the skill works offline with no server. First seen: Lesson 5
BM25
A small, standard upgrade to TF-IDF that adds length normalization and saturation. Still just arithmetic, still baked at build time. Fixes long pages winning by size. We use k1 = 1.2, b = 0.75. First seen: Lesson 6
Length normalization
Dividing a word's score by how long the page is versus the average, so a big page gets no free boost. BM25's b knob. First seen: Lesson 6
k1 (saturation)
BM25 knob that makes repeated words count for less each time — a word appearing 20× isn't 20× as relevant. First seen: Lesson 6
b (length knob)
BM25 knob (0–1) controlling how strongly to normalize by page length. b = 0.75 is the standard choice. First seen: Lesson 6
Density vs frequency
Raw frequency rewards long pages. Density (frequency relative to page length) is closer to relevance. "Frequency ≠ density ≠ relevance." First seen: Lesson 6
Code fence
A block of code or config in markdown, wrapped in triple backticks (```). Config keys, CLI flags, and file names live here. Our first indexer wrongly skipped these. First seen: Lesson 7
Tokenize
Break text into individual words ("tokens") so they can be indexed and searched. We tokenize code-fence content into a page's search text. First seen: Lesson 7
Preview snippet
The short (~500-char) human-readable summary shown next to a search result. Kept clean — code identifiers go into the search text but not into this. First seen: Lesson 7
Identifier
An exact name a platform engineer searches for: a config key, CLI flag, file name, or error string (e.g. mysqld, pg_hba.conf). The skill's core value is finding these; they live mostly in code blocks. First seen: Lesson 7
Baseline
What you compare the skill against in a test — e.g. a model with web search, or a model answering from memory only. Choosing it carefully is the whole experiment. First seen: Lesson 8
Contamination
When the thing you're testing leaks into the "control" — here a global skill in ~/.claude/skills/ that every subagent (including baselines) could use. Turns an A/B test into a meaningless A/A. First seen: Lesson 8
A/B test
Run the same questions two ways — with the skill (A) and without (B) — and compare. Only fair if B is truly without. First seen: Lesson 8
Isolate the variable
Make sure only one thing differs between A and B. Move the skill aside, hide any local docs the baseline could grep — so you measure the skill, not leakage. First seen: Lesson 8
Honest edge / overclaiming
Describe a tool by what it truly does better. The skill ties web search on correctness but wins on speed/determinism/offline — say that, not a false "more accurate." First seen: Lesson 8
AWS Docs MCP server
Amazon's documentation tool for agents. It works by giving the agent a search tool plus a fetch for live docs — the exact pattern we copied to start this design. It ships as an MCP server (software you run), which is the road we compare against in the final lesson. First seen: Lesson 1
MCP server (Model Context Protocol)
A small service the agent talks to over a standard protocol to get tools or data. Unlike a skill (a file you ship), an MCP server is software you run — hosted, or started on the user's machine (e.g. via uvx). No capability ceiling (embeddings, vector DB, live queries), but real ops cost. First seen: Lesson 1 (decided in Lesson 9)
uvx
A tool that runs a Python program on the user's machine without a manual install. One way to run an MCP server locally instead of hosting it. First seen: Lesson 9
Capability ceiling
The limit on what a package can do. A skill (static file) caps at lexical/BM25 search; an MCP server has no ceiling because it can run any code or model. First seen: Lesson 9
Client-dumb
Keeping the agent/client simple by putting the heavy work (embeddings, vector search) on a server. The MCP approach. First seen: Lesson 9
Protocol, not a corpus
The skill's consumer is an LLM, so teach it how to invoke the tool and what it's good for — not the data or the internals. Why SKILL.md stays short and references out. First seen: Lesson 9
