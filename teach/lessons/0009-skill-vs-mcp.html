Build a docs Skill · A brown-bag

Skill vs MCP server

In Lesson 1 we copied an MCP. Then we shipped a skill instead. Now you know enough to say why — and when you'd choose the other.

Why · the problem

This is where the whole course bends back on itself. We opened by copying the AWS Docs MCP pattern — search + fetch. But we never shipped an MCP server. We shipped a skill: a thin file. Why the switch? And when would the MCP have been the right call?

You've now felt the answer, lesson by lesson. Lesson 5: embeddings rank by meaning but need a model at query time. Lesson 8: for public docs the skill only ties web search. Both point at the same fork in the road. Let's name it clearly so you can make this call for your own docs.

What · the solution approach

The two shapes are the same idea — grounded search + fetch — packaged completely differently:

  • A skill is a file other agents download. It runs on the client. No server, no keys, no model to load. Dead simple to ship and to trust. The price: it can only do what fits in a static file — lexical/BM25 ranking, not embeddings.
  • An MCP server is software you run. You host it, or the user starts it locally with something like uvx. It has no capability ceiling: embeddings, a vector database (like Chroma), live queries, fancy indexing — all on the server, keeping the client "dumb." The price: someone must host it, secure it, and keep it up.
The trade in one line Skill = thin to ship, zero ops, capped capability. MCP = richer capability, real ops cost. You are trading "easy to distribute" against "can do anything."

That's exactly why embeddings (Lesson 5) live on the MCP side: they need a model at query time, so they can't ride inside a static file — they need the server an MCP gives you.

How · the mechanics

Use this table to make the call for any corpus:

QuestionSkill (a file)MCP server (you run it)
Who runs it?the client, for freeyou host, or user runs uvx
Capability ceiling?lexical / BM25 static indexembeddings, vector DB, live queries — none
Ops cost?~zero, offlinehosting, uptime, auth, updates
Freshness?re-ship the index (refresh)query live / re-embed server-side
Distribution?a file (git, a marketplace)a running endpoint + config
Best when…public/stable docs, wide reuse, offline, simpleprivate data, semantic search, central control, heavy queries

For this story the skill won cleanly: public Teleport docs, a publisher manifest, BM25 is plenty, and the value is offline speed and determinism across many agents. An MCP would have added hosting for no correctness gain. But flip to internal docs that need semantic search and change daily, and the MCP becomes the honest choice.

Whichever you ship — keep the index short (progressive discovery) The consumer is an LLM, so teach it a protocol, not a corpus. The real SKILL.md was slimmed to ~112 lines, hiding index internals behind a references/ file. As the corpus puts it: "An LLM consumer needs to know HOW to invoke it and what it's good for, not how the index is built." A short index that references out to advanced topics lets the agent discover detail only when it needs it — surfacing info in small bits. That is progressive discovery, and it applies to both skills and MCP tool descriptions.
⏱️ Worth it? — the bookend The most honest option is sometimes neither. If a model with web search already ties you on correctness (Lesson 8), and the docs are public and stable, the right call may be to skip the build and just let the agent search the web. Build a skill when offline/determinism/reuse pays for it. Build an MCP when only a server can do the job. Always weigh it against just doing the work.

The whole journey, in one breath

You built a real skill by discovering it — minimal design, test, improve:

  1. Ground answers, don't trust memory — search + fetch (Lessons 1–2).
  2. Search the body, not the label — the title-only blind spot (Lesson 3).
  3. Prove it with evals you can't rig — body-only sampling, 15–0–1 (Lesson 4).
  4. Pick what ships — TF-IDF over embeddings, for distribution (Lesson 5).
  5. Normalize — BM25 dethrones the Changelog; raw counts lie (Lesson 6).
  6. Index what people query — code blocks hold the identifiers (Lesson 7).
  7. Reality-check honestly — ties web search; edge is speed/offline (Lesson 8).
  8. Choose the package — skill vs MCP, by ops and capability (Lesson 9).

Through all of it, the same two habits: 🔴→🟢 lock every fix with a test, and ⏱️ ask if it's worth it. The deepest lesson from the corpus: the harness mattered more than the model, and a measurement — not a smarter model — caught the one real flaw.

What you can do now (back to the mission) Pick a corpus your team relies on. Check for an llms.txt. Sketch the search → fetch loop. Write three body-only evals before you build anything. Then decide, honestly: skill, MCP, or just web search? You now have the whole toolkit to make that call — and to defend it with a number.

Final self-check (recall before you peek):

1. The one-line trade between a skill and an MCP server?

Skill = thin to ship, zero ops, capped capability. MCP = richer capability (embeddings, vector DB, live), but real ops cost. Distribution vs. power.

2. Why do embeddings belong on the MCP side, not in a skill?

Searching with embeddings needs a model at query time, which can't ship in a static file — it needs the server an MCP provides.

3. What is progressive discovery, applied to a SKILL.md?

Keep the file a short index that references out to advanced topics, so the agent surfaces detail in small bits only when needed — teach a protocol, not a corpus.

4. When is the honest answer to build neither?

When web search already ties you on correctness and the docs are public and stable — skip the build unless offline/determinism/reuse pays for it.
