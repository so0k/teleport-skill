The grounding problem

Your agent just wrote the whole Teleport rollout plan — and the Terraform to match. Do you dare run terraform apply?

Why · the problem

You are not just chatting with the agent. You are planning real work with it. Picture a normal Monday:

"We're rolling out Teleport for the team and want SSO through Microsoft Entra ID using OIDC. How do I set up the OIDC auth connector, and how do I map Entra ID group claims to Teleport roles? I need the exact connector spec fields…" — eval 0, from the skill's real test set (../../evals/evals.json)

The agent writes a clean plan. It even hands you the Terraform to match (Teleport config is YAML, and the Terraform resources are very close). It looks complete. It sounds sure.

But you are not pasting a chat reply into Slack. You are about to run this as infrastructure. So the stakes are different:

  • It writes access_request.roles — but the real field is access_request.spec.roles. The plan is one word wrong.
  • It writes the Terraform role using HCL blocks, but this provider wants nested attributes with =. It won't even plan.
  • It invents a field like auto_approve that does not exist.

These are not made up. Each one is a real bug the team hit, now frozen into the test set (evals 5, 6, 8). The shape is almost right — which is worse than wrong, because a high-risk plan that looks polished is the easiest one to trust.

The grounding problem The model answers from memory — what it saw in training. But Teleport adds resource kinds, role fields, and CLI flags every release, and renames things across versions. So a remembered field name is often quietly out of date. The plan looks confident. It is stale. And you were about to build on it.

Here is the trap: the model is not dumb. It is fluent. And fluent feels like knowing.

What · the solution approach

The fix is not a smarter model. The fix is to make the agent read the docs first, ground every field in a real page, and cite that page. We call this grounding.

We did not invent the shape of this. We copied a pattern that already works: the AWS Documentation MCP server. Ask yourself — how does it ground an answer? It is simpler than you'd guess. It gives the agent just two moves:

  1. search — find the right doc page for the question.
  2. fetch — pull that live page's text, and answer from its words.

Search, then fetch. That's the whole core. The agent stops guessing field names and starts reading them. This little two-move loop is where our whole design started.

A useful frame Agent = Model + Harness. The model writes text. The harness is everything around it — the instructions, the tools, the docs it can reach, the tests that check it. That search+fetch loop is harness. As our source puts it: "the harness … matters more than the model itself."
"Agent = Model + Harness" — "the harness (context structure, tooling, evaluation) matters more than the model itself." — CORPUS.md, Act B

One thing to file away for later: AWS ships that pattern as an MCP server — software you run. We ended up shipping ours as a skill — a thin file you hand to the agent. Why we made that switch is the last lesson of this course. For now, just hold the two-move loop: search → fetch.

How · the mechanics

Same rollout question. Two ways to answer it. Watch where the field names come from.

You don't build anything yet. Keep one mechanic in your head: a grounded plan carries a source link for every field it claims. If a config can't point to a page, treat it as a guess — and never apply a guess.

That habit — "show me the page for this field" — is the seed of the whole course. In Lesson 2 we hit the obvious wall: "search" sounds like we must index 765 doc pages ourselves. Do we? (Hold that thought. The answer is a pleasant surprise.)

Quick self-check (try before you open each answer):

1. Why is a generated plan riskier than a chat reply?

You run it as infrastructure. A field that is one word wrong, or a HCL block where an attribute belongs, fails — or worse, applies the wrong thing.

2. What two moves did we borrow from the AWS Docs MCP server?

Search (find the right doc page) and fetch (pull that live page's text and answer from it). Search, then fetch.

3. Fill in: Agent = Model + ______. Which part does our search+fetch loop change?

Harness. The loop is part of the harness — the work around the model — not the model itself.
