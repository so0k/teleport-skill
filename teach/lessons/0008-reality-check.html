Build a docs Skill · A brown-bag

The reality check

We built a great search. But did we just build a slower Google?

Why · the problem

By Lesson 7 the search is genuinely good: body, BM25, code blocks, all guarded by evals. Time to ask the question we've been ducking.

A plain model with web search can read these same Teleport docs — Google indexed them long ago. So is our skill actually better? Or did we spend all this effort rebuilding something the model could already reach?

To answer honestly, we run an A/B test: the same questions, once with the skill, once without. Simple. Except "without" is where two nasty traps hide — and falling into either gives you a confident, wrong answer.

The honest goal We are not trying to prove the skill wins. We are trying to find out the truth. If the honest answer is "it barely beats web search," we want to know that — and say it.
What · the solution approach

Trap 1 — a contaminated baseline. The first A/B was a lie. The skill lived in ~/.claude/skills/, where it's visible to every subagent — including the "baseline" ones. They had a pushy description, so the baselines quietly used the skill too. We compared the skill… to itself.

"those baselines are really with-skill runs and must be redone." — CORPUS.md, §4 (the contamination crisis)

The fix: isolate the variable. Physically move the skill aside, and hide the local docs checkout one baseline was caught grepping. Only then is "without" really without.

Trap 2 — the baseline you pick decides the result. This is the deep one:

  • Compare against a web-enabled model and the skill ties — because that model just looks up the same public docs. You've tested nothing about grounding; both sides are grounded.
  • Compare against a training-only model (no tools, "Claude in the wild") and a real gap appears — the bare model invents schema it can't verify from memory.
Choosing the baseline IS the experiment The number you get depends entirely on who you race against. So pick the baseline that isolates the exact property you claim. If you claim "grounding," race the model that can't ground. If you claim "beats web search," race web search — and accept the answer.

Your turn — predict before the reveal.

We race the finished skill against a model that can use web search over the same public Teleport docs. On correctness, what happens — and what's left for the skill to win on?

Reveal the result

Correctness ties: 13/13 vs 13/13. The web model reads the same docs through Google, so it's just as correct. What's left is how: the skill is ~29% faster (76s vs 107s) and fully offline/deterministic. The edge is speed and reliability — not raw capability.

How · the mechanics

Same skill, two opponents, two completely different verdicts:

"13/13 vs 13/13 — identical correctness to a web_search baseline, but the skill is 29% faster (76s vs 107s mean) … the skill's edge is speed/determinism/offline-search, not raw capability." — CORPUS.md, Act I (the honest final read)

So we describe the skill by its true edge. Not "more accurate than the web" — that's false. But "as accurate, and faster, deterministic, and works offline" — that's true, and genuinely useful.

⏱️ Worth it? — the honest verdict For public docs Google already indexed, the skill barely beats web search on correctness — its payoff is speed/determinism/offline. That may or may not be worth the build, depending on reuse. But flip the corpus: for internal / private docs Google can't see, there is no web baseline — grounding is the whole game, and the skill (or a server) is the only honest path. Same technique, very different ROI. Know which world you're in.

Here's the thread pulling into our last lesson. For public docs the skill ties web search. Embeddings might rank better — but they need a server. So when is shipping a skill even the right call… and when should you host an MCP server instead?

Quick self-check (recall before you peek):

1. How did a global skill contaminate the baseline?

It lived in ~/.claude/skills/, visible to every subagent, so the "baseline" runs secretly used it too — an A/A test dressed up as A/B.

2. Why does "choosing the baseline IS the experiment"?

3. What is the skill's honest edge vs web search on public docs?

Not correctness — that ties. Speed, determinism, and offline search (~29% faster). For internal docs with no web baseline, grounding itself is the edge.
