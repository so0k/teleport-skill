The title-only blind spot

Your search reads one line per page. What if the word you need never shows up in that line?

Why · the problem

In Lesson 2 we got a cheap win: search the llms.txt manifest instead of crawling the site. But remember what a manifest line actually holds — just a title and a one-line description:

- [PostgreSQL](/docs/.../postgres-self-hosted.md): Enroll a self-hosted PostgreSQL database with Teleport.

Now read a real question a platform engineer asks (this is eval 13 from the test set):

"I'm enrolling a self-hosted Postgres database and need it to require client certificates. What pg_hba.conf settings and hostssl entry do I need?" — eval 13, ../../evals/evals.json

The right page is that PostgreSQL page. But look: the words that make this question specific — pg_hba.conf, hostssl, "client certificates" — appear nowhere in the title or the description. They live deep in the page body.

So a search that only reads manifest lines never matches. The perfect page scores zero. We call that a complete miss — rank 0.

The blind spot A title-only search can only find a page by words in its title or one-line description. But the most useful questions are full of words that live only in the body — error strings, config keys, file names, CLI flags. Exactly the identifiers your grounded plan depends on. The search is blind to all of them.
"search matched only the one-line title+description per page — 'no page bodies, so it's cheap … an index lookup, not a content search.' This blind spot wouldn't surface until evals." — CORPUS.md, Act A (the latent flaw)

This is sneaky because the cheap version feels fine on easy questions. It breaks on exactly the hard, specific ones — and those are the ones you most need grounded.

What · the solution approach

The fix is plain to say: stop searching only the one-line summary. Search the whole page body. Index every word in the markdown twin, not just the title.

Two names for the two approaches — keep them straight, we use them all course:

ApproachWhat it indexesFinds pg_hba.conf?
Title-only (what we have)title + one-line descriptionNo — rank 0
Full-text / content (what we want)the entire page bodyYes — rank 1

It sounds obvious once said. The catch is cost: now we must download all 765 markdown twins, pull out their words, and store that. That's a real index — the thing we happily avoided in Lesson 2. We'll build it properly in Lessons 5 and 6. For now, just hold the idea: search the body, not the label.

⏱️ Worth it? Before you build that index, ask: how often will this skill be used, and by how many agents? A full-text index is real, ongoing work. It only pays off if the blind spot actually bites your users often enough to matter. We'll measure exactly that in the next lesson — so the decision is grounded too, not a hunch.
How · the mechanics

Same question, two indexes. Watch what each one can "see."

Here's the part that matters for you as a skill-builder. We just claimed "title-only misses, full-text finds." That's a nice story. But how do we know it's true — and true often enough to be worth the cost of a real index?

We could eyeball a handful of queries and nod. But eyeballing is vibes, not evidence. And a few lucky examples can hide a real pattern. To trust the fix, we have to measure it — on many questions, fairly.

That raises a juicy problem: where do we get a pile of fair, body-only questions — ones whose answer words live only in the page body — without hand-writing them one by one? In Lesson 4 we generate them with a workflow, then run the benchmark. (Spoiler from the corpus: the score was lopsided.)

Your turn — spot the body-only query.

The PostgreSQL page is titled "PostgreSQL", described "Enroll a self-hosted PostgreSQL database." Which question would a title-only search still find, and which is a body-only blind spot? Decide, then check.

A. "How do I enroll a self-hosted PostgreSQL database?"

Title-only finds it. The words "enroll", "self-hosted", "PostgreSQL" are right there in the title and description. Easy question, easy hit.

B. "Which pg_hba.conf hostssl entry requires client certs?"

Body-only blind spot. None of pg_hba.conf / hostssl / "client certs" appear in the label — title-only scores it rank 0, even though it's the same page.

Recall check:

What is a "complete miss" (rank 0)?

When the correct page shares no words with the label the search reads, so it never matches — the right answer scores zero.

See it for yourself (primary source): Open Teleport's real llms.txt and find the PostgreSQL line — note it has no pg_hba.conf. Then open that page's .md twin and use your browser's find (⌘F) for hostssl: it's only in the body. That gap is the blind spot.
