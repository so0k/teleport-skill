We build one running example, lesson by lesson: the real teleport-docs skill. Each lesson is a 10-minute win and leads into the next. Every claim is grounded — in the project's own story (../provenance/CORPUS.md) or a trusted source (RESOURCES.md).

Two threads run through every lesson Agile iteration — we don't design the perfect skill up front; we ship a minimal version, test it, and improve, so each round sources more accurate data than the last. And a 🔴→🟢 / ⏱️ worth-it habit — every fix gets locked in by a test, and we keep asking whether the ancillary work beats just doing the job. The honest ending is allowed to be: "this barely beats a web search."
(A separate idea, progressive disclosure / discovery, shows up in Lesson 2 and again at the end: the skill surfaces info in small bits — a short index that references out.)

The arc

  1. 1 The grounding problem Planning rollouts with an agent is high-risk. The AWS Docs MCP pattern: search → fetch. Agent = Model + Harness. Ready
  2. 2 Borrow the publisher's index You don't index 765 pages — Teleport already did, via llms.txt + clean .md twins. Ready
  3. 3 The title-only blind spot Searching one line per page misses answers buried in the body. How would you even know? Ready
  4. 4 Evals that sample the blind spot Mine "body-only" questions with a workflow; benchmark the search. The 15–0–1 result. Ready
  5. 5 Pick the engine: why not embeddings? Inkeep and vectors rejected — for shippability. TF-IDF is just arithmetic. Ready
  6. 6 Raw counts lie → BM25 The Changelog out-ranks the real answer by sheer size. Length normalization (k1, b). Ready
  7. 7 Index what people query mysqld and pg_hba.conf were invisible — they live in code blocks. Ready
  8. 8 Reality check vs. a Google-indexed model Ties on correctness, wins on speed/determinism/offline. Be honest about the edge. Ready
  9. 9 Step back: skill vs. MCP server We copied an MCP — then shipped a skill. Hosting vs. a thin file. When each wins. Ready
